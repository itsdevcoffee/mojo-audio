# mojo-audio Roadmap

**Last updated:** 2026-03-06
**Context:** Building the DSP + ML inference layer for a local, offline voice conversion
pipeline (Shade) that runs natively on NVIDIA DGX Spark SM_121 ARM64.

---

## North Star

> The entire Shade voice conversion pipeline — stem separation → content extraction →
> pitch estimation → synthesis → output — runs end-to-end on DGX Spark with GPU acceleration,
> zero PyTorch CUDA dependency, zero cloud.

## Sprint 2 — "GPU Pipeline Complete" (~2–3 weeks after Sprint 1)

**Goal:** All stages that CAN run on GPU DO run on GPU. Full pipeline is functional
if slow (VITS still in Python for now).

### 2A. pos_conv GPU Fix

Two paths depending on Modular's response:
- **If MAX is patched:** Remove numpy bridge, test full GPU inference, re-benchmark
- **If not patched:** Implement pos_conv as custom Mojo op registered with MAX
  (`max.torch.graph_op` decorator pattern, or write a native Mojo kernel)

Expected result: AudioEncoder GPU drops from 197ms → ~15–30ms on Spark (matching
the 16x GPU speedup we saw in isolated matmul benchmarks).

### 2B. Phase-Locked Phase Vocoder

Current OLA pitch shifter produces phasiness on sustained tones. Replace with proper
phase vocoder that tracks inter-frame phase continuity.

- Algorithm: STFT → accumulate phase with corrected advance per bin → ISTFT
- The complex STFT helper (`_stft_complex`) already exists in `audio_encoder.py` —
  expose it in the DSP layer and wire into pitch.mojo
- Expected quality improvement: significant, especially for piano and sustained vocals

### 2C. AudioEncoder Batch > 1

Current implementation is hardcoded `[1, ...]`. Change graph shapes to `[B, ...]`
for parallel multi-user inference on Spark.

### 2D. MAX Graph Kernel Cache Verification

First `from_pretrained()` call takes ~168s due to JIT compilation. Verify the kernel
cache is persistent across Python sessions (it should be, per MAX docs) and that our
graph naming scheme (`"audio_encoder"`) generates stable cache keys.

### Definition of Done — Sprint 2

- [ ] AudioEncoder GPU (x86): < 10ms for 1s audio
- [ ] AudioEncoder GPU (Spark): < 20ms for 1s audio
- [ ] Phase shifter quality confirmed on sustained tone test
- [ ] Batch=4 inference works

## Sprint 3 — "HiFiGAN Vocoder" (~3–4 weeks)

**Goal:** Implement the neural vocoder in MAX Graph. This is the first half of VITS
synthesis — it takes mel spectrograms and produces audio. Once done, we can do
mel-spectrogram → audio on Spark without PyTorch.

### Why HiFiGAN first

VITS is two components: (1) the encoder/decoder that converts features to mel specs,
(2) HiFiGAN which converts mel specs to waveforms. HiFiGAN alone is useful because:
- It's the bottleneck in any vocoder pipeline (most CPU-intensive step)
- It's purely convolutional — standard ops, cleaner to implement than VITS decoder
- Combined with our existing DSP layer (mel spectrogram via Mojo), we can already do
  mel→audio entirely without PyTorch on Spark

**Architecture:** Multiple upsampling blocks with residual dilated convolutions.
All standard `ops.conv2d`, `ops.conv_transpose`, `ops.leaky_relu` — no exotic ops.

### Definition of Done — Sprint 3

- [ ] HiFiGAN inference in MAX Graph, numerically matches PyTorch ± 0.01 on mel input
- [ ] Runs on Spark GPU
- [ ] Mel → audio round-trip test passes (generate → mel → HiFiGAN → audio)

## Sprint 4 — "Full VITS Synthesis" (~4–6 weeks)

**Goal:** Complete VITS decoder in MAX Graph. Combined with HuBERT + RMVPE,
the full RVC-style voice conversion pipeline runs on Spark.

### VITS components to implement

- Posterior encoder (converts waveform to latent)
- Normalizing flows (transforms latent distribution)
- Decoder (latent → mel spectrogram, feeds into HiFiGAN)
- Stochastic duration predictor (less critical for inference)

### Architecture risks

VITS is significantly more complex than HuBERT:
- ~95M params (same as HuBERT) but more heterogeneous ops
- Normalizing flow layers use custom invertible transforms
- Duration predictor uses stochastic sampling (needs to work at inference)

**Mitigation:** Start with ONNX export → check op coverage → implement missing ops
as Mojo custom ops → full MAX Graph rewrite. The ONNX path won't work for loading
(MAX has no ONNX importer) but is useful for validating which ops we need.

### Definition of Done — Sprint 4

- [ ] VITS inference in MAX Graph matches PyTorch output ± 1e-3
- [ ] RVC voice conversion: ContentVec features + RMVPE F0 + VITS → converted audio
- [ ] Runs end-to-end on DGX Spark with GPU
- [ ] Full integration test: WAV in → WAV out, matches Applio quality

## Sprint 6 — "Multi-Spark" (Post-Shade Launch)

**Goal:** NVLink tensor parallelism across paired Sparks.

### What to implement

- Shard transformer layers across 2 NVLinked Sparks using `max.nn.DistributedTransformer`
- Target: VITS synthesis sharded across 2 GPUs (256GB unified memory pool)
- Enable batch processing of multiple voice conversions simultaneously

### When to do this

After Shade is working on a single Spark. The multi-Spark work is for scale —
handling concurrent requests from multiple clients. For the initial demo, single-Spark
is sufficient.
