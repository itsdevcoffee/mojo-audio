## Item 2B — Phase-Locked Phase Vocoder

**Priority: Start here. Highest quality impact, completely unblocked.**

### What the current OLA does (and why it's wrong)

The current implementation is in `src/pitch.mojo`, function `_time_stretch_ola` (lines 51–106). It uses plain overlap-add: it takes input frames, applies a Hann window, and drops them into the output buffer at stretched positions. This is the "identity phase vocoder" — it moves frames around but does **nothing** about phase continuity between frames.

The problem: adjacent STFT frames have phases that need to track each other across the stretch. When you move a frame from position A to position B without adjusting its phase, you get destructive interference at the overlap boundaries. This produces the characteristic phasiness artifact on sustained tones (piano, held vocals).

### What phase-locking fixes

A proper phase vocoder tracks the **instantaneous frequency** of each bin across frames and propagates the phase correctly:

```
For each STFT frame f and each frequency bin k:
  1. Compute phase difference: Δφ_k = angle(X_f[k]) - angle(X_{f-1}[k])
  2. Compute expected phase advance: φ_expected = 2π * k * hop_length / n_fft
  3. Compute deviation: δ_k = Δφ_k - φ_expected  (wrapped to [-π, π])
  4. Compute instantaneous frequency: ω_k = (φ_expected + δ_k) / hop_length
  5. Propagate synthesized phase: ψ_f[k] = ψ_{f-1}[k] + ω_k * out_hop
  6. Reconstruct: Y_f[k] = |X_f[k]| * exp(iψ_f[k])
```

Then IFFT each Y_f and OLA into the output buffer.

### Implementation plan

The existing `audio.mojo` already has:
- `hann_window(n_fft)` — already imported in pitch.mojo
- `stft(signal, n_fft, hop_length, window_fn)` — returns `List[List[Float32]]` (power spectrum, not complex)
- `_stft_complex` — exists in `audio_encoder.py` as a Python helper, but NOT in `audio.mojo` as a Mojo function

**The gap:** `audio.mojo` exposes only power spectrum from STFT, not complex STFT output (magnitude + phase). The phase vocoder needs the full complex STFT.

**Approach:**
1. Add a `stft_complex` function to `audio.mojo` that returns complex-valued frames as `List[List[Complex]]` (real + imaginary per bin). The existing FFT infrastructure (`fft`, `rfft`, complex types) is already there.
2. Replace `_time_stretch_ola` in `pitch.mojo` with `_time_stretch_pv` — a proper phase vocoder using the complex STFT
3. Keep the same `pitch_shift` public API — no external interface changes

**Key Mojo facts for this codebase:**
- Complex type: `Complex[DType.float32]` with `.re` and `.im` fields
- `atan2` is available via `math`
- `math.constants.pi` for π
- Wrap phase to `[-π, π]`: subtract `2π * round(x / 2π)`
- The existing `fft()` in `audio.mojo` returns `List[Complex]`

## Item 2C — AudioEncoder Batch > 1

**Priority: Second. ~2–3 days, straightforward.**

### What's hardcoded

`AudioEncoder._from_weights` in `src/models/audio_encoder.py` builds two MAX Graphs with batch=1 baked in:

- **Graph 1** (`cnn_proj`): `TensorType(DType.float32, [1, Dim("L"), 1, 1], device_ref)` — line 164
- **Graph 2** (`encoder_transformer`): `TensorType(DType.float32, [1, Dim("T"), 768], device_ref)` — line 230
- All reshape ops inside use literal `1` for batch: `ops.reshape(x, [1, -1, 512])` etc.
- The numpy pos_conv bridge in `encode()` also assumes `[1, T, 768]` shape

### What needs to change

1. Add `batch_size: int = 1` parameter to `_from_weights` and `from_pretrained`
2. Replace all hardcoded `1` batch dims with the `batch_size` parameter in graph construction
3. Update the pos_conv numpy bridge in `encode()` to handle `[B, T, 768]` (the `for grp in range(groups)` loop processes `x_1t = features_np[0]` — needs to loop over batch)
4. Update `encode()` signature to accept `[B, N]` audio input (currently expects `[1, N]`)
5. Update `test_audio_encoder.py` — add batch=2 shape tests to `TestAudioEncoderShapes`

### Key things to watch

- `_perm4` operates on 4D tensors — should be batch-agnostic already
- The pos_conv numpy bridge processes one sample at a time in a group loop — will need an outer `for b in range(B)` loop

## Item 2D — MAX Kernel Cache Verification

**Priority: Last. Investigation only, ~2–3 hours. No implementation.**

### Background

The first call to `AudioEncoder.from_pretrained()` or `PitchExtractor.from_pretrained()` takes ~168 seconds due to MAX Graph JIT compilation. This should be cached on disk across Python sessions. If it isn't, every cold start is 168s which is unacceptable for production.

### What to verify

1. Time the first `from_pretrained()` call in a fresh Python process
2. Exit and time again in a new Python process
3. If the second call is fast (<5s), the cache works — document it
4. If the second call is also slow, find where MAX stores its cache (check `~/.cache/modular/`, `~/.local/share/modular/`, or env var `MODULAR_CACHE_DIR`) and investigate whether our graph naming scheme (`"cnn_proj"`, `"encoder_transformer"`, `"rmvpe_unet"`) generates stable cache keys across sessions

### Investigation script

```python
# experiments/max-cache-check/check_kernel_cache.py
import time
from models import AudioEncoder

print("Loading AudioEncoder (may take ~168s first run)...")
t0 = time.time()
model = AudioEncoder.from_pretrained("facebook/hubert-base-ls960", device="cpu")
t1 = time.time()
print(f"Load time: {t1-t0:.1f}s")
print("Run this script twice. Second run should be <5s if cache works.")
```

### Definition of done

- Document whether cache works (yes/no) and where it lives
- If broken: file an issue / find workaround
- Update `docs/context/` with findings
